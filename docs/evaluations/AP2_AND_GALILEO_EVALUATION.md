## Executive Summary

**Google Agent2Agent (A2A) Protocol** is an open standard (Apache 2.0, Linux Foundation) for inter-agent communication using JSON-RPC 2.0 over HTTP(S). It provides agent discovery via Agent Cards, a stateful task lifecycle model, artifact exchange, streaming (SSE), and push notifications (webhooks). It is explicitly designed to complement MCP rather than replace it -- MCP handles tool/context integration while A2A handles opaque agent-to-agent delegation. SDKs exist for Python, Go, JS, Java, .NET. Enterprise partners include Google Cloud, Salesforce, SAP, Atlassian, MongoDB, and others.

**Galileo** is an AI observability, evaluation, and production guardrail platform. It accepts telemetry via OpenTelemetry/OTLP, provides 20+ out-of-box evaluation metrics (RAG, agentic, safety, hallucination, SQL), custom LLM-as-a-judge evaluators, and "Luna" distilled models for low-latency/low-cost production guardrails. Pricing starts free (5K traces/month), Pro at $100/month (50K traces), Enterprise custom. Python and TypeScript SDKs with integrations for LangChain, OpenAI, CrewAI, and Google ADK.

## Part 1: Google Agent2Agent (A2A) Protocol

### 1.1 Protocol Overview

- **Transport**: JSON-RPC 2.0 over HTTP(S), with gRPC and HTTP/REST bindings also specified
- **Normative source**: `spec/a2a.proto` (Protocol Buffers), with JSON schema generated at build time
- **Version**: Latest spec at [a2a-protocol.org/latest/specification/](https://a2a-protocol.org/latest/specification/)
- **License**: Apache 2.0 under Linux Foundation
- **SDKs**: Python (`pip install a2a-sdk`), Go, JS (`npm install @a2a-js/sdk`), Java (Maven), .NET (NuGet)

### 1.2 Agent Discovery (Agent Cards)

Agents publish JSON metadata documents ("Agent Cards") describing:
- Identity and service endpoint URL
- Capabilities (streaming, push notifications, extended agent card)
- Skills (what the agent can do)
- Authentication requirements and security schemes
- Accepted/output content types

Agent Cards can be extended post-authentication via `GetExtendedAgentCard` to reveal additional skills to authorized clients.

### 1.3 Task Lifecycle Model

Tasks are the fundamental unit of work. State machine:

```
submitted -> working -> completed
                     -> failed
                     -> canceled
                     -> rejected
         -> input-required (multi-turn)
         -> auth-required
```

Key properties:
- **Server-generated task IDs** (clients cannot propose IDs)
- **Context IDs** group related tasks into conversational sessions
- **Multi-turn**: Tasks can transition to `input-required`, prompting the client to send follow-up messages
- **Blocking vs non-blocking**: `return_immediately` flag controls whether send-message waits for completion
- **History**: Configurable `historyLength` parameter controls how much conversation history is returned

### 1.4 Artifact Exchange Format

Artifacts are structured outputs generated by agents:
- Composed of **Parts**: text, file references (URIs), or structured JSON data
- Messages also contain Parts with roles ("user" or "agent")
- File parts support any MIME type
- Streaming artifact updates via `TaskArtifactUpdateEvent`
- Content type negotiation via `acceptedOutputModes` in request configuration

### 1.5 Push Notification Mechanism

For long-running tasks without persistent connections:
- Client registers a webhook URL via `CreateTaskPushNotificationConfig`
- Agent sends HTTP POST to webhook with `StreamResponse` payloads (status updates, artifact updates)
- CRUD operations for notification configs (create, get, list, delete)
- Uses plain HTTP + JSON regardless of the protocol binding used by the agent
- Requires `AgentCard.capabilities.pushNotifications: true`

### 1.6 Enterprise Partners

The A2A protocol has broad enterprise backing. Google Cloud customers can join a partner program. The protocol is contributed by Google to the Linux Foundation. Known enterprise supporters include companies building SDKs and integrations across the major cloud and SaaS ecosystem.

### 1.7 A2A vs MCP Comparison

| Dimension | MCP | A2A |
|-----------|-----|-----|
| **Purpose** | Tool/context integration (agent-to-tool) | Agent-to-agent delegation |
| **Opacity** | Client sees tool schemas, calls tools directly | Agents are opaque; no internal state exposure |
| **State** | Stateless tool calls | Stateful tasks with lifecycle |
| **Discovery** | Server capabilities via `initialize` | Agent Cards with skills, auth, capabilities |
| **Transport** | JSON-RPC over stdio/HTTP+SSE | JSON-RPC, gRPC, HTTP/REST |
| **Streaming** | SSE for notifications | SSE + push notifications (webhooks) |
| **Multi-turn** | Not native | `input-required` state, context IDs |
| **Relationship** | Complementary -- MCP is the "tool layer" | Complementary -- A2A is the "agent layer" |

### 1.8 Fit Assessment

#### 1.8.1 trvl -- Multi-Provider Travel Search Orchestration

**Fit: GOOD (with caveats)**

trvl currently operates as a single MCP server with 41 tools that scrape/query multiple travel data sources internally (Google Flights, Google Hotels, 20+ ground transport providers, Ticketmaster, etc.). The internal architecture is already multi-provider via the `internal/providers/` package.

A2A could enable:
- **External agent delegation**: A travel orchestrator agent could delegate "find flights HEL-NRT" to a flights-specialist A2A agent, "find hotels in Tokyo" to a hotel agent, etc.
- **Multi-turn booking flows**: A2A's `input-required` state maps well to booking confirmation flows ("Found 3 options, which do you want to book?")
- **Third-party agent integration**: If Booking.com, Skyscanner, or airline agents expose A2A endpoints, trvl could consume them directly

However:
- trvl's strength is that it already integrates multiple providers internally with zero API keys required (scraping). A2A adds value only when integrating with *external* agentic systems, not for internal provider orchestration
- The current MCP interface already works well for the primary use case (Claude/AI assistant as the orchestrator)
- A2A would be more relevant if trvl wanted to *expose itself* as an A2A-compliant agent to other systems

**Recommendation**: EVALUATE FURTHER -- implement A2A server mode for trvl (expose trvl as an A2A agent) before building A2A client capabilities. The Go SDK (`github.com/a2aproject/a2a-go`) makes this straightforward.

**Integration effort**: M (Medium) -- 2-3 weeks for A2A server wrapper around existing MCP tools

#### 1.8.2 mcp-gateway -- A2A as Transport Alongside MCP

**Fit: STRONG**

mcp-gateway already has a layered architecture with protocol types (`src/protocol/`), transport layer (`src/transport/` with HTTP and stdio), and backend management (`src/backend/`). The gateway's core value proposition is abstracting multiple backend servers behind a unified Meta-MCP surface.

An A2A adapter would:
- Allow the gateway to **discover and proxy A2A agents** alongside MCP backends
- Map A2A Agent Cards to the gateway's capability/tool discovery system
- Translate between MCP tool calls and A2A `SendMessage` operations
- Expose gateway-managed tools to external A2A clients (gateway as A2A server)

Architecture sketch for an A2A adapter:

```
src/
  a2a/
    mod.rs           -- A2A module root
    agent_card.rs    -- AgentCard parsing, discovery, caching
    client.rs        -- A2A JSON-RPC client (send message, get task, stream)
    backend.rs       -- A2ABackend impl (implements Backend trait)
    server.rs        -- Expose gateway as A2A server (Agent Card, endpoints)
    types.rs         -- A2A protocol types (Task, Message, Part, Artifact)
    transport.rs     -- HTTP transport for A2A (reuse reqwest::Client)
```

Key mapping:
| MCP Concept | A2A Equivalent |
|-------------|----------------|
| `tools/call` | `message/send` with task tracking |
| `tools/list` | Agent Card skills |
| Tool result (text/json) | Task artifact (Parts) |
| Server capabilities | Agent Card capabilities |

The gateway could support A2A backends in `config.yaml`:

```yaml
backends:
  - name: travel-agent
    transport: a2a
    url: https://travel.example.com/.well-known/agent.json
    auth: bearer
```

**Recommendation**: ADOPT -- A2A support would differentiate mcp-gateway as a true multi-protocol gateway, not just MCP-to-MCP routing. The A2A spec is stable enough (Linux Foundation governance) to build against.

**Integration effort**: L (Large) -- 4-6 weeks for full bidirectional A2A support (client + server). M (Medium) if limited to A2A-as-backend-only (3 weeks).

#### 1.8.3 Botnaut -- Enterprise Payment Rails

**Fit: PARTIAL**

A2A provides the agent-to-agent communication substrate but does not define payment-specific primitives. For enterprise payment rails:
- A2A's **task lifecycle** maps well to payment authorization flows (submitted -> working -> input-required [for 2FA/approval] -> completed)
- A2A's **opaque execution** model is appropriate for financial agents that must not expose internal logic
- A2A's **multi-tenant** support (`tenant` field on all operations) aligns with enterprise multi-tenancy
- A2A's **security model** (OAuth2, API keys, JWT) is enterprise-grade

However:
- No built-in payment primitives (amounts, currencies, settlement, reconciliation)
- No financial compliance framework (PCI-DSS, KYC/AML)
- Payment-specific semantics would need to be encoded as A2A Extensions or structured JSON Parts
- Botnaut would need to define its own payment protocol on top of A2A transport

**Recommendation**: EVALUATE FURTHER -- A2A is a viable transport layer for Botnaut's agent-to-agent payment negotiations, but the payment protocol itself must be custom-built. Consider A2A for the inter-agent communication channel while building domain-specific payment primitives in the message/artifact layer.

**Integration effort**: L (Large) -- A2A transport + custom payment protocol on top

## Part 2: Galileo AI

### 2.1 Platform Overview

Galileo is an AI observability and evaluation platform focused on:
1. **Telemetry ingestion** -- traces, spans, sessions via SDK or OpenTelemetry/OTLP
2. **Evaluation metrics** -- 20+ out-of-box metrics across 7 categories
3. **Guardrails** -- distilled "Luna" models for production-time content filtering
4. **Experiments** -- systematic A/B testing of prompts, models, and configurations
5. **Insights engine** -- automated failure mode detection and fix recommendations

### 2.2 Telemetry Format

Galileo accepts telemetry in two primary formats:

**1. Native SDK (Python/TypeScript)**:
- Hierarchical: Project -> Log Stream -> Session -> Trace -> Spans
- `GalileoLogger` with explicit `start_trace()` / `add_span()` / `conclude_trace()` / `flush()`
- Span types: LLM call, tool call, workflow step, retriever, etc.
- Auto-instrumentation via decorators/wrappers

**2. OpenTelemetry/OTLP**:
- Standard OTLP endpoint: `https://api.galileo.ai/otel/traces`
- `GalileoSpanProcessor` handles auth and metadata automatically
- OpenInference semantic conventions for AI-specific attributes (`gen_ai.request.model`, `gen_ai.response.content`, `gen_ai.usage.*`)
- Integrations: LangGraph, OpenAI, Google ADK, Strands Agents, Vercel AI SDK

### 2.3 Evaluation Metrics

Seven metric categories, all configurable per Log Stream:

| Category | Key Metrics |
|----------|-------------|
| **Agentic Performance** | Tool selection accuracy, task completion, goal progress |
| **Expression & Readability** | Tone, clarity, reading level, format adherence |
| **Model Confidence** | Uncertainty scoring, low-confidence flagging |
| **Response Quality** | Correctness, instruction adherence, ground truth adherence |
| **RAG** | Chunk relevance, context precision, context adherence, completeness, attribution |
| **Safety & Compliance** | PII detection, prompt injection, bias, harmful content |
| **Text-to-SQL** | Query correctness, schema grounding, injection prevention |

Custom metrics supported via:
- LLM-as-a-Judge (custom evaluation prompts)
- Code-based metrics (arbitrary Python/TS functions)
- **Autotune**: Continuous feedback loop that improves metric alignment with domain expertise

### 2.4 Hallucination Detection

Galileo's hallucination detection operates at multiple levels:
- **Context Adherence** (RAG): Does the response stay grounded in retrieved context?
- **Chunk Attribution**: Can each claim be traced to a specific retrieved chunk?
- **Ground Truth Adherence**: Does the response match known-correct answers?
- **Correctness**: General factual accuracy scoring
- Luna-2 models provide low-latency hallucination scoring distilled from expensive LLM judges

### 2.5 Guardrails as a Service

Galileo's "Protect" offering:
- Distills expensive LLM-as-judge evaluators into compact **Luna** models
- Runs on L4 GPUs with low latency for production use
- Monitors 100% of traffic at claimed 97% lower cost vs full LLM evaluation
- Eval scores can control: agent actions, tool access, escalation paths
- Pre-production evals become production governance automatically
- Deployment: SaaS, VPC, or on-premises

| **Free** | $0 | 5,000 | Unlimited users, unlimited custom evals |
| **Pro** | $100/mo | 50,000 | Standard RBAC, advanced analytics, Slack support |

### 2.7 API Documentation

- Auth: API key header, HTTP Basic, or JWT Bearer token
- SDK: `pip install galileo` (Python), `npm install galileo` (TypeScript)
- OpenTelemetry: Standard OTLP exporter with `GalileoSpanProcessor`

### 2.8 Fit Assessment

#### 2.8.1 mcp-gateway -- Consuming cost_accounting Telemetry

**Fit: MODERATE (with mapping work)**

mcp-gateway's `cost_accounting` module tracks:
- Per-session costs: `SessionCostSnapshot` with call_count, total_tokens, total_cost_usd, by_backend, by_tool breakdowns
- Per-API-key costs: `KeyCostSnapshot` with rolling windows (24h/7d/30d), budget limits, by_tool breakdown
- Individual `CostRecord` events: timestamp, backend, tool, token_count, estimated_cost_usd
- Cost governance: configurable budgets, alerts at 50%/80%/100%, per-tool costs via `CostRegistry`
- Persistence: `costs.json` with cumulative `ToolTotal` (call_count, total_cost_usd, avg_cost_usd)

Galileo integration path:
1. **OTLP export**: Add a `GalileoSpanProcessor`-compatible exporter that maps each `CostRecord` to an OTLP span with cost attributes
2. **Span mapping**: Each gateway tool call becomes a Galileo span with:
   - `gen_ai.usage.input_tokens` / `gen_ai.usage.output_tokens`
   - Custom attributes for `estimated_cost_usd`, `backend`, `budget_status`
3. **Session mapping**: Gateway MCP sessions map to Galileo sessions
4. **Metrics**: Galileo's agentic performance metrics could evaluate tool selection quality; cost data would need custom code-based metrics

Gaps:
- Galileo expects LLM-centric spans (prompt/response); mcp-gateway's cost data is tool-call-centric
- The gateway's rolling-window budget model has no direct Galileo equivalent
- Cost governance alerts (50/80/100% thresholds) would need to remain in the gateway; Galileo adds observability, not enforcement

**Recommendation**: EVALUATE FURTHER -- Start with OTLP export of tool-call spans to get Galileo's visualization and metric capabilities. Defer deep integration until there's a concrete need for Galileo's hallucination detection or guardrails on gateway-routed responses.

**Integration effort**: S (Small) for OTLP span export (~1 week). M (Medium) for full metric integration with custom evaluators (~3 weeks).

#### 2.8.2 mcp-gateway -- Galileo as a Backend

**Fit: WEAK**

Galileo is an observability/evaluation platform, not a capability provider. It does not expose tool-like functionality that would map to MCP tools. The integration direction is:
- mcp-gateway **sends data to** Galileo (telemetry export)
- NOT: mcp-gateway **calls** Galileo as a backend

The only scenario where Galileo-as-backend makes sense is if the gateway wanted to expose Galileo's evaluation APIs as MCP tools (e.g., `galileo_evaluate_response`, `galileo_check_hallucination`). This would require:
- A new capability YAML for Galileo tools
- A custom backend that proxies to Galileo's REST API
- Rate limiting awareness (Galileo API quotas)

**Recommendation**: SKIP as a backend. ADOPT as a telemetry sink / observability layer.

**Integration effort**: N/A (not recommended as backend)
